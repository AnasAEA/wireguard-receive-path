\begin{abstract}
%% ============================================================

WireGuard is a modern VPN built into the Linux kernel. Under sustained
multi-gigabit traffic with 1,000 concurrent clients, it reaches only 19.2\% of
available bandwidth. Prior work traced this to an Execution Order Inversion
(EoI): concurrent decryption triggers Generic Receive Offload (GRO) reassembly
so frequently that the CPU saturates. That work moved reassembly to a background thread and recovered
4.7$\times$ throughput.

We show, through source-level analysis, that this fix is incomplete. It makes
each wasted reassembly cheaper, but not less frequent. WireGuard schedules
reassembly after \emph{every} decrypted packet, even when no progress is
possible. With decryption spread over $N$ cores, the packet due for delivery
next is rarely the first to finish, so most reassembly passes find nothing to
deliver. We propose a six-line conditional guard: reassembly is triggered only
when the next packet due is ready. We prove correctness under concurrent access
and evaluate the fix on an ARM platform (Apple M1~Pro).

\end{abstract}

\section{Introduction}
%% ============================================================

WireGuard~\cite{donenfeld2017} is a modern VPN integrated into the Linux kernel
since version~5.6 (2020). It uses ChaCha20-Poly1305 encryption~\cite{rfc8439}
and the Noise protocol framework~\cite{noise}. Its minimal codebase and
competitive performance have made it a common replacement for older VPN
solutions such as IPsec and OpenVPN.

At multi-gigabit throughput, however, WireGuard exhibits unexpected performance
degradation on multi-core systems. \citeauthor{mounah2025}~\cite{mounah2025}
measured a throughput collapse to 19.2\% of line rate and 94\% CPU utilization
under 25~Gbps with 1,000 clients. They identified an Execution Order Inversion
(EoI) and proposed moving GRO from a high-priority softirq (a deferred
interrupt handler) to a normal-priority workqueue (a pool of background kernel
threads), recovering 4.7$\times$ throughput.

Despite this result, the fix is incomplete. Moving GRO lowers the cost of each
wasted poll (an invocation of the reassembly function that delivers nothing) but
does not lower their count. The root cause is unchanged: WireGuard calls
\texttt{napi\_schedule} (the function that requests a GRO poll) after every
decrypted packet regardless of queue state. GRO can only deliver if the \emph{head} of the per-peer queue is
already decrypted. With $N$ packets decrypting concurrently the head finishes
first with probability $1/N$, so on average $(N\!-\!1)/2$ polls per batch find
nothing to deliver. The problem is in the trigger logic, not the scheduling
priority.

This work presents a source-level analysis and a targeted fix: a six-line guard
at the scheduling site that invokes GRO only when the head is ready. We prove
the guard correct under concurrent access and evaluate it experimentally.

\smallskip\noindent This paper addresses the following research questions:
\begin{description}
  \item[RQ1.] What is the exact mechanism that causes GRO to be invoked
    wastefully in WireGuard's receive path?
  \item[RQ2.] Does the prior fix eliminate the root cause or only reduce
    per-invocation cost?
  \item[RQ3.] Can a conditional guard eliminate the dominant source of wasted
    invocations without breaking correctness or packet ordering?
  \item[RQ4.] Does the problem manifest on ARM as on x86-64?
\end{description}

\smallskip\noindent\textbf{Scope and limitations.} We focus on the data receive
path. The handshake path uses a separate, blocking workqueue and is not
addressed here. Measurements are on ARM (Apple M1~Pro, Fedora Asahi Remix);
x86-64 validation on dedicated server hardware (CloudLab, 25~Gbps NIC) is in
progress.

\section{Background}
\label{sec:background}
%% ============================================================

\subsection{WireGuard's Receive Pipeline}

WireGuard's receive path~\cite{wireguard-linux} has three stages. Stage~1
runs in bottom-half (BH) context: it de-encapsulates incoming UDP datagrams
and dispatches them for decryption. Stage~2 performs ChaCha20-Poly1305
decryption inside a \emph{workqueue}, a pool of kernel threads running at
normal scheduler priority~\cite{linux-workqueue}. Stage~3 reassembles
decrypted packets and passes them to the network stack through Generic Receive
Offload (GRO), executing as a \emph{softirq}, a kernel routine that runs at
higher priority than normal threads and cannot sleep.

Two separate queues serve different purposes (Figure~\ref{fig:queues}):
\begin{itemize}
  \item The \emph{per-peer RX queue} (\texttt{prev\_queue}) holds one peer's
    packets in arrival order to fix \emph{delivery order}.
  \item The \emph{global device ring} (\texttt{ptr\_ring}) distributes packets
    across CPU cores by round-robin to pick \emph{which core} decrypts each one.
\end{itemize}
Delivery order is fixed first, when the packet enters the per-peer queue with
state \textsc{uncrypted}. When a worker finishes, it flips the state to
\textsc{crypted} (or \textsc{dead} on failure).

\begin{figure}[t]
  \centering
  \includegraphics[width=\linewidth]{report_two_queues.pdf}
  \caption{The same packet enters both queues for different reasons. The
  per-peer queue fixes \emph{delivery order}; the device ring picks \emph{which
  CPU} decrypts. Workers flip each packet's state; \texttt{wg\_packet\_rx\_poll}
  delivers from the head in order.}
  \label{fig:queues}
\end{figure}

\subsection{NAPI, GRO, and Softirqs}

NAPI (New API) avoids per-packet interrupts by batching~\cite{linux-napi}:
after the first packet arrives, the kernel disables the hardware interrupt and
instead processes packets in groups. Each receive queue is paired with a poll function; WireGuard
creates one per peer, using \texttt{wg\_packet\_rx\_poll}. To request a poll,
code calls \texttt{napi\_schedule}, which marks the queue as ``scheduled'' and
raises a software interrupt. This does not run the poll immediately; it only
schedules it to run shortly after. Repeated calls while the queue is already
marked collapse into a single pending poll.

The poll runs shortly after \texttt{napi\_schedule} returns, at the next point
where the worker briefly re-enables interrupt handling (a call to
\texttt{spin\_unlock\_bh} at the end of each loop iteration). When it fires,
\texttt{wg\_packet\_rx\_poll} walks the per-peer queue from its head and passes
packets to GRO, which merges consecutive packets of the same flow to amortize
the stack-traversal cost. The poll stops at the first \textsc{uncrypted} entry
and handles at most 64 packets per run (the poll budget).
\texttt{napi\_complete\_done} then clears the scheduled flag.

\subsection{Linux Workqueues}

Workqueues defer work to normal-priority background threads. WireGuard's decrypt
workqueue (\texttt{packet\_crypt\_wq}) uses \texttt{WQ\_PERCPU}, provisioning
one worker per online core. Work items are dispatched to a specific core via
\texttt{queue\_work\_on}. Workers run at \texttt{SCHED\_NORMAL} and can be
preempted by softirqs at any BH re-enable point, which is why decryption cannot
run in the softirq directly.

\section{Related Work}
\label{sec:sota}
%% ============================================================

\textbf{The Execution Order Inversion.}
\citeauthor{mounah2025}~\cite{mounah2025} first identified and named the EoI in
WireGuard's receive path. They diagnose it as a priority conflict: the GRO poll
runs at high softirq priority and preempts the workers that are producing the
packets it needs. Their fix moves GRO execution to a dedicated workqueue running
at normal priority, eliminating the preemption and recovering $4.7\times$
throughput. An alternative variant using dedicated kernel threads achieves a
similar result with lower tail latency. Both approaches lower the \emph{cost} of
each empty poll but leave the unconditional trigger unchanged, so empty polls
continue to fire at the same rate. Our work targets the other half: reducing the
\emph{count} of empty polls by guarding the trigger. The two fixes are
orthogonal and can be combined.

\textbf{Kernel bypass approaches.}
A different strategy for high-speed packet I/O is to bypass the kernel stack
altogether. netmap~\cite{rizzo2012netmap} exposes raw NIC rings to userspace,
eliminating per-packet system call overhead. The eXpress Data Path
(XDP)~\cite{hoiland2018xdp} runs eBPF programs at the earliest possible point
in the kernel, before memory allocation, reaching near-wire speed on commodity
hardware. DPDK takes the furthest step: a dedicated poll-mode driver that
removes interrupt overhead entirely. These approaches trade generality and
security isolation for throughput. WireGuard deliberately stays in the kernel to
inherit the security guarantees of the kernel networking stack and interoperate
with the rest of the system. The EoI is a consequence of this in-kernel design,
and fixing it in-kernel preserves these properties.

\textbf{Kernel asynchronous I/O.}
\texttt{io\_uring}~\cite{axboe2019} is the kernel's general-purpose
asynchronous I/O interface, also designed to reduce context-switch and
system-call overhead through submission and completion rings shared between
kernel and userspace. \citeauthor{mounah2025} connect WireGuard's workqueue
model directly to this lineage: their fix can be seen as applying the
asynchronous dispatch pattern to the GRO stage. Our fix targets a different
layer, improving the scheduling policy within that asynchronous stage rather
than its execution context.

\section{Root Cause Analysis}
\label{sec:rootcause}
%% ============================================================

Three conditions interact to produce the overhead: delivery is gated by the
queue head, decryption finishes out of order across cores, and the GRO trigger
fires unconditionally regardless of whether the head is ready or not.

\subsection{Delivery Is Gated by the Head}

Each packet carries a state (\texttt{PACKET\_CB(skb)->state}, an
\texttt{atomic\_t}): \textsc{uncrypted} on enqueue, \textsc{crypted} on
success, \textsc{dead} on failure. \texttt{wg\_packet\_rx\_poll} walks from the
head and delivers while state is not \textsc{uncrypted}, halting at the first
\textsc{uncrypted} entry (\texttt{receive.c:451}). No packet can be delivered
while the head is still pending, regardless of how many later packets are ready.

\subsection{Decryption Finishes Out of Order}

Consecutive packets from one peer are not decrypted together. The device ring
assigns each to a core by round-robin (\texttt{queueing.h:120}), so they
decrypt in parallel and complete in an order set by per-core load, not arrival.
Figure~\ref{fig:eoi} shows the result: a later packet finishes first and
schedules a poll, but the head is still pending.

\begin{figure}[t]
  \centering
  \includegraphics[width=0.92\linewidth]{report_eoi.pdf}
  \caption{Execution Order Inversion. Packet~2 finishes first and schedules a
  poll, but the head (packet~0) is still \textsc{uncrypted}, so the poll returns
  \texttt{work\_done = 0}.}
  \label{fig:eoi}
\end{figure}

\subsection{The Trigger Ignores Both}

When a worker finishes, it calls \texttt{napi\_schedule} unconditionally
(\texttt{queueing.h:195--196},~\cite{wireguard-linux}):

\begin{lstlisting}[language=C]
atomic_set_release(&PACKET_CB(skb)->state, state);
napi_schedule(&peer->napi);
\end{lstlisting}

Each of $N$ parallel completions raises a poll, but only the one that makes
the head ready delivers anything. Every other completion produces a
\texttt{work\_done = 0} poll. The decrypt worker then proceeds to the next
packet and calls \texttt{napi\_schedule} again, so the wasted polls recur for
as long as the head lags behind.

\subsection{Why the Waste Grows with Core Count}

Among $N$ packets decrypting concurrently, the head finishes first with
probability $\approx 1/N$. As $N$ grows, more siblings complete ahead of the
head, and a larger fraction of polls find it still \textsc{uncrypted}. The
trigger fires once per decrypted packet; a poll makes progress only at the head.
The more cores, the more wasted polls. We measure the actual rate in
Section~\ref{sec:eval}.

%% ============================================================
\section{Proposed Fix}
\label{sec:fix}
%% ============================================================

\subsection{Condition Design}

The fix schedules the poll only when the front of the per-peer queue is no
longer \textsc{uncrypted}. If the front is still pending the wake is skipped;
the worker that eventually makes the front ready will wake the poll then.

Reading the front from worker context requires care. The standard accessor
\texttt{wg\_prev\_queue\_peek} is reserved for the poll (the queue's single
consumer) because it caches its result; calling it from a worker would corrupt
that state. Instead we read the consumer's position directly from
\texttt{peer->rx\_queue.tail}. Because this field is written only by the
consumer (\texttt{queueing.c:87,92,101}), a worker can safely read it: a
slightly stale value is harmless because the check is only a hint: if we
mistakenly skip a wake, the worker that eventually makes the head ready will
schedule the poll. If \texttt{tail} points to the stub sentinel (a dummy node
the queue uses to mark the empty state), we schedule unconditionally, which
costs at most one extra poll.

\subsection{The Change}

Six lines in \texttt{queueing.h}, replacing the unconditional call:

\begin{lstlisting}[language=C]
/* was: napi_schedule(&peer->napi); */
tail = READ_ONCE(peer->rx_queue.tail);
if (tail == (struct sk_buff *)
            &peer->rx_queue.empty ||
    atomic_read(&PACKET_CB(tail)->state) !=
            PACKET_STATE_UNCRYPTED)
        napi_schedule(&peer->napi);
\end{lstlisting}

\subsection{Correctness}

If the finishing packet is the front, the front is now non-\textsc{uncrypted},
the guard passes, and the poll finds work. If the front is already
\textsc{crypted}/\textsc{dead} the guard passes. If the front is still
\textsc{uncrypted} the guard correctly skips the wake. For stale reads: if this
worker misses a concurrent flip to \textsc{crypted}, the core that made that
flip runs the same guard and schedules the poll, so no packet is stranded. The
read can be relaxed (\texttt{atomic\_read}, no memory barrier): we are only
deciding whether to wake, not consuming decrypted data, so no stronger ordering
guarantee is needed.

NAPI also handles the case where \texttt{napi\_schedule} is called while a poll
is already running: rather than dropping the request, NAPI records an internal
\texttt{MISSED} flag and runs the poll one more time once the current run
finishes (\texttt{dev.c:6744,6825}). This ensures no delivery is lost; at worst
it is deferred by one extra poll.

\subsection{Residual Limitation}

The guard does not reduce empty polls to exactly zero. The reason is the
\texttt{MISSED} mechanism that makes it safe. Suppose packets~0 and~1 finish
decrypting at almost the same time:
\begin{enumerate}
  \item The worker for packet~0 sees the front ready and wakes the poll, which
    starts running.
  \item The poll delivers packet~0, then packet~1 (also ready), and stops at
    packet~2, which is still \textsc{uncrypted}.
  \item The worker for packet~1 also saw the front ready and called
    \texttt{napi\_schedule}, but the poll was already running, so this only set
    \texttt{MISSED}.
  \item When the poll finishes, \texttt{MISSED} is set, so NAPI re-runs it.
    Packet~1 is already delivered and packet~2 is not ready, so this extra run
    finds nothing.
\end{enumerate}
One empty poll can remain per such overlap. In the unmodified code, every
out-of-order completion produced an empty poll and the count grew with core
count; here, only these occasional \texttt{MISSED} re-runs remain, and they do
not scale with concurrency.

\section{Evaluation}
\label{sec:eval}
%% ============================================================

\subsection{Experimental Setup}

\textbf{Platform.} Apple M1~Pro (ARM64, 10 cores), Fedora Asahi Remix~42,
kernel \texttt{6.19.13-400.asahi.fc44.aarch64+16k}. The patched module exposes
sysfs knobs: \texttt{wg\_eoi\_fix} (0\,=\,stock, 1\,=\,fix) and
\texttt{wg\_decrypt\_delay\_us} (artificial per-packet busy-wait). Stock and
patched are compared within the same module instance and tunnel, eliminating
variance from module reloads.

\textbf{Topology and traffic.} We use Linux network namespaces (isolated
virtual network stacks within the same kernel) to simulate $N$ independent
clients on a single machine: one server namespace and $N$ client namespaces,
each holding a separate WireGuard peer with its own key pair. iperf3 drives a
fixed total of 32 streams split across $N$ peers ($\lfloor 32/N \rfloor$
streams/peer), holding offered load constant across the peer sweep. Each cell
runs 30~s with a 5~s warm-up omitted.

\textbf{Variance control and metrics.} Each (peers, build) cell is measured
5~times; stock and patched alternate order each run to cancel drift. GRO poll
counts and batch size (\texttt{work\_done}, the number of packets delivered per
poll pass) are collected with bpftrace (a Linux kernel dynamic tracing tool)
probes on \texttt{wg\_packet\_rx\_poll}. Latency is from 20~Hz ping
(p50/p99/p99.9). To reduce variance from the M1's asymmetric cores, the
efficiency cores are taken offline before each run (\emph{ecores mode}).

\subsection{Results: Regime Test}
\label{subsec:regime}

The peer sweep confirms the fix works on this rig, but leaves one question
open: does it hold when decryption is the actual bottleneck, not just a fast
background task? On the M1, ChaCha20 is so fast that workers never saturate
their cores, so the EoI causes waste but not throughput collapse. To approach
the regime where it does, we slow decryption artificially (40~µs per packet via
\texttt{wg\_decrypt\_delay\_us}) and restrict to 2~CPU cores. This forces
workers and the softirq to compete for the same cores, mimicking a heavily
loaded server with slower hardware.

The fix still reduces wasted polls by 19.6\% (Table~\ref{tab:regime}),
confirming that the guard works regardless of decrypt speed. Throughput is
essentially unchanged because at 40~µs/packet and only 2 cores, decrypt
capacity is the bottleneck regardless of GRO efficiency. The full
throughput-collapse observed by \citeauthor{mounah2025} requires a real NIC
driving traffic faster than the workers can process it, which even artificial
delay on a loopback cannot reproduce.

\subsection{Discussion}

Together, the two experiments answer all four research questions. The mechanism
is confirmed: \texttt{napi\_schedule} fires unconditionally at
\texttt{queueing.h:196}, and \texttt{wg\_packet\_rx\_poll} returns
\texttt{work\_done = 0} whenever the head is still \textsc{uncrypted}, which
happens systematically when $N > 1$ workers decrypt in parallel (RQ1). The prior fix
by \citeauthor{mounah2025} addresses this by running GRO at lower priority, but
by source inspection their patch leaves the unconditional \texttt{napi\_schedule}
call unchanged, so empty polls are still scheduled at the same rate (RQ2). Our
guard reduces those polls by 9--22\% across peer counts, with batch size rising
in every case, a direct sign that GRO is now woken at moments where it can
actually coalesce packets (RQ3). Finally, the EoI manifests on ARM with 25--31\%
of polls wasted under stock, and since \texttt{queueing.h} and \texttt{receive.c}
are identical line-for-line between the Asahi kernel and the x86 v6.1 kernel
used by \citeauthor{mounah2025}, the mechanism and the fix transfer directly
(RQ4).

\subsection{The Cost of Each Wake}
\label{subsec:costwake}

The latency increase in Table~\ref{tab:tput} points to a trade-off inherent in
the fix. By waiting until the head is ready before waking, the poll may deliver
only one packet when it runs. WireGuard then pays the full overhead of a
softirq, a GRO pass, and a NAPI completion to deliver a single packet, with
none of GRO's batching benefit. The head packet also arrives slightly later
than it would have under an unconditional wake.

Whether this is a real problem depends on two costs we have not yet measured:
how expensive a poll is, and how much it costs to deliver and copy a packet to
the application. If delivery dominates, waking for one packet is fine. If poll
overhead dominates, a policy that waits briefly for more packets would be
better. We chose the simple wake-on-ready policy because it keeps latency
bounded and never withholds a ready packet, which is the right default for a
VPN. Measuring these two costs on real hardware will determine whether this
simple policy is good enough long-term, or whether a batching-aware trigger is
needed. We leave this as the primary open question.

\section{Conclusion}
\label{sec:conclusion}
%% ============================================================

WireGuard's EoI stems from an unconditional \texttt{napi\_schedule} call after
every decrypted packet: GRO polls systematically find nothing to deliver when
the per-peer queue head is still \textsc{uncrypted}, and the fraction of such
polls grows with core count. A six-line conditional guard reads
\texttt{READ\_ONCE(peer->rx\_queue.tail)} before waking the poll and skips the
wake when the head is not ready. We proved correctness under stale reads and
the stub sentinel edge case. Evaluation on ARM (Apple M1~Pro) confirms a
9--22\% reduction in wasted GRO polls across peer counts 1--32 at fixed total
load, and a 20\% reduction under artificial heavy-crypto conditions with 2-core
contention. Batch size increases in both cases, confirming the mechanism.
Throughput is unchanged on this loopback rig, as expected: the benefit appears
in poll-efficiency metrics, not Gbps.

The fix by \citeauthor{mounah2025} reduces the cost of each GRO invocation;
ours reduces how many invocations happen. The two are independent, and applying
both together should give additional improvement over either alone.

\smallskip\noindent\textbf{Future work.}
(1)~Validate the throughput effect on x86-64 hardware with a real 25~Gbps NIC
(CloudLab), the regime in which \citeauthor{mounah2025} measured.
(2)~Combine the conditional guard with the paper's workqueue-priority fix.
(3)~Measure wake cost vs.\ packet-delivery cost, then design a batching-aware
trigger policy that wakes only when waking pays off.
(4)~Extend the analysis to the handshake path, which uses a separate blocking
workqueue and may exhibit related overheads.
