\begin{abstract}
%% ============================================================

WireGuard, a modern VPN integrated into the Linux kernel, achieves only
19.2\% of available bandwidth under sustained multi-gigabit traffic with
1,000 concurrent clients. Prior work identified an Execution Order Inversion
(EoI)---a structural conflict between WireGuard's packet decryption pipeline
and its packet reassembly scheduling mechanism---and proposed moving packet
reassembly to a dedicated background thread as a fix, recovering 4.7$\times$
throughput. Through source-level analysis, we show that this fix reduces the
cost of each wasted reassembly invocation but does not reduce their number.
Under concurrent decryption across $N$ CPU cores, the probability that a
reassembly invocation finds useful work is at most $1/N$: on an 8-core
machine, 87.5\% of invocations are structurally wasted. We propose a
six-line conditional check at the scheduling site that eliminates the dominant
source of wasted invocations, prove its correctness under concurrent access,
and evaluate it on an ARM platform (Apple M1~Pro).

\end{abstract}

\section{Introduction}
%% ============================================================

Virtual Private Networks (VPNs) are essential infrastructure for secure
communication in enterprise and cloud environments. WireGuard~\cite{donenfeld2017}
is a modern VPN implementation integrated into the Linux kernel since
version~5.6 (2020). It is built around simplicity and correctness: a minimal
codebase, the ChaCha20-Poly1305 authenticated encryption
algorithm~\cite{rfc8439}, and a cryptographic handshake based on the Noise
protocol framework~\cite{noise}. Its straightforward design and competitive
performance have made it the preferred replacement for older VPN solutions
such as IPsec and OpenVPN.

At multi-gigabit throughput, however, WireGuard exhibits unexpected
performance degradation on multi-core systems.
\citeauthor{mounah2025}~\cite{mounah2025} measured a throughput collapse to
approximately 19.2\% of line rate and 94\% CPU utilization under sustained
25~Gbps traffic with 1,000 concurrent clients. The authors identified the
source of this degradation as an interaction between WireGuard's concurrent
packet decryption pipeline and the Linux kernel's Generic Receive Offload
(GRO) scheduling mechanism---a phenomenon they term an Execution Order
Inversion (EoI). Their proposed fix, which moves GRO execution from a
high-priority software interrupt context to a dedicated background thread
running at normal scheduling priority, yields a 4.7$\times$ throughput
improvement~\cite{mounah2025}.

Despite this result, the fix is incomplete. Moving GRO to a background thread
reduces the cost of each scheduling event but does not reduce the number of
such events. The root cause---WireGuard unconditionally schedules GRO after
every decrypted packet, regardless of whether the packet queue is in a state
where GRO can make any progress---is left unaddressed. Under concurrent
decryption across $N$ CPU cores, the probability that GRO finds useful work
on any given invocation is at most $1/N$. On an 8-core machine, at least
87.5\% of GRO invocations are structurally wasted. This is not a scheduling
priority problem; it is a trigger logic problem.

This work presents a source-level analysis of WireGuard's receive pipeline
and a targeted fix for the root cause. The analysis traces the exact sequence
of events that leads to wasted GRO invocations, identifies the structural
conditions under which they occur, and provides a probabilistic model
predicting their frequency as a function of CPU count. The proposed fix adds
a six-line conditional check at the GRO scheduling site: GRO is invoked only
when the head of the per-peer receive queue is ready for processing. We prove
this check correct under concurrent access and evaluate it experimentally.

\smallskip\noindent This paper addresses the following research questions:
\begin{description}
  \item[RQ1.] What is the exact mechanism that causes GRO to be invoked
    wastefully in WireGuard's receive path?
  \item[RQ2.] Does the fix by \citeauthor{mounah2025} eliminate the root
    cause of the overhead, or only reduce its per-invocation cost?
  \item[RQ3.] Can a conditional check at the scheduling site eliminate the
    dominant source of wasted invocations without breaking correctness or
    packet ordering?
  \item[RQ4.] Does the problem manifest with the same intensity on ARM as
    on x86-64?
\end{description}

\smallskip\noindent\textbf{Scope and limitations.} We focus on WireGuard's
data receive path. The handshake path uses a separate, blocking workqueue and
is not addressed here. Measurements are conducted on ARM (Apple M1~Pro,
Fedora Asahi Remix); a cross-architecture comparison on x86-64 or RISC-V via
QEMU is left as future work.

\section{Background}
\label{sec:background}
%% ============================================================

\subsection{WireGuard's Receive Pipeline}

WireGuard's receive path consists of three ordered stages. Stage~1 runs
inside the kernel's UDP receive handler, which executes in \emph{bottom-half}
(BH) context---a deferred, high-priority execution mode triggered by hardware
interrupts. It de-encapsulates incoming UDP datagrams and dispatches them for
decryption. Stage~2 performs ChaCha20-Poly1305 decryption inside a
\emph{workqueue}: a kernel-managed pool of background threads that run at
normal scheduler priority. Stage~3 reassembles decrypted packets and hands
them to the network stack via the Generic Receive Offload (GRO) mechanism,
which executes as a \emph{softirq} (a high-priority deferred interrupt
handler).

Packets flow through two structurally independent queues. The \emph{global
device ring} (\texttt{ptr\_ring}) holds packets from all peers mixed together;
it determines which CPU core decrypts each packet, assigned by approximate
round-robin across online cores. The \emph{per-peer RX queue}
(\texttt{prev\_queue}) holds packets from a single peer in strict arrival
order; it determines the order in which packets are delivered to the network
stack. These two roles are separated by design: delivery order is established
when a packet is inserted into the per-peer queue (Phase~1, state set to
\textsc{uncrypted}), before it is dispatched to any CPU (Phase~2). A worker
atomically upgrades the state to \textsc{crypted} or \textsc{dead} upon
completing decryption.

\subsection{NAPI, GRO, and Softirqs}

The Linux NAPI interface is a polling mechanism for network receive
processing. Each WireGuard peer has one \texttt{napi\_struct} instance.
Calling \texttt{napi\_schedule} raises the \texttt{NET\_RX\_SOFTIRQ} software
interrupt and sets an internal scheduling flag; a second call while the flag
is set is a no-op. When the softirq fires, WireGuard's poll function
\texttt{wg\_packet\_rx\_poll} walks the per-peer RX queue from its head,
delivers all contiguous non-\textsc{uncrypted} packets to the network stack
via GRO, and stops at the first \textsc{uncrypted} entry. The poll budget is
64~packets (the kernel default). When the poll completes without exhausting
the budget, \texttt{napi\_complete\_done} clears the scheduling flag,
re-arming the NAPI instance for future invocations.

The softirq fires at the earliest \texttt{spin\_unlock\_bh} call following
\texttt{napi\_schedule}---in WireGuard's case, at the next iteration of the
decrypt worker's packet consumption loop.

\subsection{Linux Workqueues}

Workqueues are the kernel's standard mechanism for deferring work to
normal-priority background threads. WireGuard's decrypt workqueue
(\texttt{packet\_crypt\_wq}) is created with the \texttt{WQ\_PERCPU} flag,
which provisions one worker thread per online CPU core. Work items are
dispatched to a specific core via \texttt{queue\_work\_on(cpu, wq, work)}.
Workers run at \texttt{SCHED\_NORMAL} (standard Linux scheduling priority)
and can be preempted by softirqs at any BH re-enable point.

\subsection{Two Independent Queues}

% Global device ring mixes packets from ALL peers; determines which CPU decrypts each packet
% Per-peer RX queue maintains strict arrival order; determines delivery order to the network stack

% wg_packet_rx_poll can only deliver packets in order: it walks from tail (oldest) forward
%

\subsection{Concurrent Decryption}

% wg_cpumask_next_online: approximate round-robin across online CPUs (queueing.h:115-119)
% Same-peer packets dispatched to different CPUs concurrently -- confirmed at source
% Example: 4 packets from Peer A → CPU 0, CPU 1, CPU 2, CPU 3 in parallel
% CPU 2 finishes first → pkt 2 is CRYPTED, pkts 0,1 still UNCRYPTED
% GRO looks at pkt 0 (head) → UNCRYPTED → returns work_done=0. Pkt 2 is unreachable.
%
% Head packet has systematic disadvantage: CPU 0 may be finishing a previous peer's packet
%   when pkt 0 arrives, so it starts later than CPUs 1,2,3 start their packets.
%
% Figure: \ref{fig:concurrent}

\subsection{The Unconditional Trigger}

% wg_queue_enqueue_per_peer_rx (queueing.h:188-198):
%   atomic_set_release(&PACKET_CB(skb)->state, state);  // mark CRYPTED
%   napi_schedule(&peer->napi);                         // unconditional
%
% napi_schedule raises NET_RX_SOFTIRQ at next spin_unlock_bh in decrypt worker's loop
% wg_packet_rx_poll (receive.c:451): while loop stops at first UNCRYPTED
% If head is UNCRYPTED: work_done=0, napi_complete_done clears SCHED flag
% Worker resumes, processes next packet, calls napi_schedule again -- cycle repeats
%
% Figure: \ref{fig:loop}

\subsection{Probabilistic Model}

% For GRO to deliver packet N (at position N in peer->rx_queue):
%   All packets at positions 0..N must be CRYPTED before GRO runs.
%   The packet at position 0 must finish first.
%
% Assuming uniform decryption times: P(pkt 0 finishes first among N concurrent) = 1/N
%   N=2: 50% wasted   N=4: 75% wasted   N=8: 87.5% wasted   N=16: 93.75% wasted
%
% Two structural amplifiers:
%   1. Head packet starts later (CPU 0 may be busy) -- makes 1/N an upper bound, not average
%   2. Trigger fires per-packet, not per-batch -- every completion fires a separate napi_schedule
%
% Cumulative effect: the 94% CPU saturation measured by Mounah et al.

\subsection{Condition Design}

% GRO makes progress if and only if: head of peer->rx_queue is NOT UNCRYPTED
% Simplified trigger condition: skip napi_schedule when head is UNCRYPTED
%
% Implementation constraint: cannot call wg_prev_queue_peek from worker context
%   (single-consumer guarantee -- would corrupt peeked cache)
% Safe alternative: READ_ONCE(peer->rx_queue.tail)
%   - tail written only by consumer (wg_prev_queue_dequeue, queueing.c:87,92,101)
%   - reading from worker context is safe as a hint (no write, no cache corruption)
%   - STUB sentinel: tail == &peer->rx_queue.empty → schedule conservatively

\subsection{Correctness}

% Four cases after packet N is marked CRYPTED:
%   Case A: N is head, now CRYPTED → tail state non-UNCRYPTED → schedule. GRO finds work.
%   Case B: N not head, head already CRYPTED/DEAD → schedule. GRO finds work.
%   Case C: N not head, head UNCRYPTED → skip. GRO would find nothing.
%   Case D: N not head, gap before N → skip (subcase of C, head is UNCRYPTED).
%   STUB:   Queue on sentinel boundary → schedule conservatively.
%
% Stale read safety: if worker reads UNCRYPTED when head just became CRYPTED,
%   it skips napi_schedule. The CPU that flipped the head reads CRYPTED and schedules.
%   No packet is stranded. Liveness preserved.
%
% atomic_read (not atomic_read_acquire): relaxed read correct -- this is a hint, not
%   a synchronization point. Avoids unnecessary memory barrier.
%
% Figure: \ref{fig:fix}

\subsection{Residual Limitation}

% Narrow timing window: GRO delivers pkts 0..1, stops at pkt 2 (UNCRYPTED), work_done=2.
%   Between GRO stopping and napi_complete_done clearing NAPI_STATE_SCHED:
%   CPU 2 marks pkt 2 CRYPTED, checks tail → CRYPTED → calls napi_schedule.
%   NAPI_STATE_SCHED still set → napi_schedule is a no-op.
%   napi_complete_done clears the flag. No one reschedules.
%   Pkts 2..N wait until next packet arrives and re-triggers the condition.
%
% Not a correctness issue. Under sustained high-throughput traffic, the next packet
%   arrives in nanoseconds. Under bursty traffic, this may manifest as a rare tail
%   latency spike.
