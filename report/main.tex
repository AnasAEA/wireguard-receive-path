\begin{abstract}
%% ============================================================

WireGuard is a modern VPN built into the Linux kernel. Under sustained
multi-gigabit traffic with 1,000 concurrent clients, it reaches only 19.2\% of
available bandwidth. Prior work traced this to an Execution Order Inversion
(EoI): concurrent decryption triggers GRO reassembly so often that the CPU
saturates. That work moved reassembly to a background thread and recovered
4.7$\times$ throughput.

We show, through source-level analysis, that this fix is incomplete. It makes
each wasted reassembly cheaper, but not less frequent. WireGuard schedules
reassembly after \emph{every} decrypted packet, even when no progress is
possible. With decryption spread over $N$ cores, the packet due for delivery
next is rarely the first to finish, so most reassembly passes find nothing to
deliver. We propose a six-line conditional guard: reassembly is triggered only
when the next packet due is ready. We prove correctness under concurrent access
and evaluate the fix on an ARM platform (Apple M1~Pro).

\end{abstract}

\section{Introduction}
%% ============================================================

WireGuard~\cite{donenfeld2017} is a modern VPN integrated into the Linux kernel
since version~5.6 (2020). It uses ChaCha20-Poly1305 encryption~\cite{rfc8439}
and the Noise protocol framework~\cite{noise}. Its minimal codebase and
competitive performance have made it a common replacement for older VPN
solutions such as IPsec and OpenVPN.

At multi-gigabit throughput, however, WireGuard exhibits unexpected performance
degradation on multi-core systems. \citeauthor{mounah2025}~\cite{mounah2025}
measured a throughput collapse to 19.2\% of line rate and 94\% CPU utilization
under 25~Gbps with 1,000 clients. They identified an Execution Order Inversion
(EoI) and proposed moving GRO from a high-priority softirq to a normal-priority
workqueue, recovering 4.7$\times$ throughput.

Despite this result, the fix is incomplete. Moving GRO lowers the cost of each
wasted poll but does not lower their count. The root cause is unchanged:
WireGuard calls \texttt{napi\_schedule} after every decrypted packet regardless
of queue state. GRO can only deliver if the \emph{head} of the per-peer queue
is already decrypted. With $N$ packets decrypting concurrently the head
finishes first with probability $1/N$, so on average $(N\!-\!1)/2$ polls per
batch find nothing to deliver. The problem is in the trigger logic,
not the scheduling priority.

This work presents a source-level analysis and a targeted fix: a six-line guard
at the scheduling site that invokes GRO only when the head is ready. We prove
the guard correct under concurrent access and evaluate it experimentally.

\smallskip\noindent This paper addresses the following research questions:
\begin{description}
  \item[RQ1.] What is the exact mechanism that causes GRO to be invoked
    wastefully in WireGuard's receive path?
  \item[RQ2.] Does the prior fix eliminate the root cause or only reduce
    per-invocation cost?
  \item[RQ3.] Can a conditional guard eliminate the dominant source of wasted
    invocations without breaking correctness or packet ordering?
  \item[RQ4.] Does the problem manifest on ARM as on x86-64?
\end{description}

\smallskip\noindent\textbf{Scope and limitations.} We focus on the data receive
path. The handshake path uses a separate, blocking workqueue and is not
addressed here. Measurements are on ARM (Apple M1~Pro, Fedora Asahi Remix);
x86-64 validation on dedicated server hardware (CloudLab, 25~Gbps NIC) is in
progress.

\section{Background}
\label{sec:background}
%% ============================================================

\subsection{WireGuard's Receive Pipeline}

WireGuard's receive path has three stages. Stage~1 runs in bottom-half (BH)
context: it de-encapsulates incoming UDP datagrams and dispatches them for
decryption. Stage~2 performs ChaCha20-Poly1305 decryption inside a
\emph{workqueue}, a pool of kernel threads running at normal scheduler
priority. Stage~3 reassembles decrypted packets and passes them to the network
stack through Generic Receive Offload (GRO), executing as a \emph{softirq}.

Two separate queues serve different purposes (Figure~\ref{fig:queues}):
\begin{itemize}
  \item The \emph{per-peer RX queue} (\texttt{prev\_queue}) holds one peer's
    packets in arrival order to fix \emph{delivery order}.
  \item The \emph{global device ring} (\texttt{ptr\_ring}) distributes packets
    across CPU cores by round-robin to pick \emph{which core} decrypts each one.
\end{itemize}
Delivery order is fixed first, when the packet enters the per-peer queue with
state \textsc{uncrypted}. When a worker finishes, it flips the state to
\textsc{crypted} (or \textsc{dead} on failure).

\begin{figure}[t]
  \centering
  \includegraphics[width=\linewidth]{report_two_queues.pdf}
  \caption{The same packet enters both queues for different reasons. The
  per-peer queue fixes \emph{delivery order}; the device ring picks \emph{which
  CPU} decrypts. Workers flip each packet's state; \texttt{wg\_packet\_rx\_poll}
  delivers from the head in order.}
  \label{fig:queues}
\end{figure}

\subsection{NAPI, GRO, and Softirqs}

NAPI batches packet delivery: after the first packet arrives, the kernel stops
per-packet interrupts and polls the device in batches instead. Each receive
queue has a \texttt{napi\_struct} and a poll function; WireGuard assigns each
peer its own, with \texttt{wg\_packet\_rx\_poll} as its poll function. Calling
\texttt{napi\_schedule} sets a ``scheduled'' flag and raises
\texttt{NET\_RX\_SOFTIRQ}; it does not run the poll immediately. If
\texttt{napi\_schedule} is called again while the flag is still set, the second
call has no effect: repeated requests collapse into a single pending poll.

The softirq fires at the next BH re-enable point (\texttt{spin\_unlock\_bh}),
which in the decrypt worker's loop is the next iteration. When it fires,
\texttt{wg\_packet\_rx\_poll} walks the per-peer queue from its head and passes
packets to GRO, which merges consecutive packets of the same flow to amortize
the cost of traversing the network stack. The poll stops at the first
\textsc{uncrypted} entry and handles at most 64 packets per run (the poll
budget). \texttt{napi\_complete\_done} then clears the scheduled flag.

\subsection{Linux Workqueues}

Workqueues defer work to normal-priority background threads. WireGuard's decrypt
workqueue (\texttt{packet\_crypt\_wq}) uses \texttt{WQ\_PERCPU}, provisioning
one worker per online core. Work items are dispatched to a specific core via
\texttt{queue\_work\_on}. Workers run at \texttt{SCHED\_NORMAL} and can be
preempted by softirqs at any BH re-enable point, which is why decryption cannot
run in the softirq directly.

\section{Related Work}
\label{sec:sota}
%% ============================================================

The work closest to ours is by \citeauthor{mounah2025}~\cite{mounah2025}, who
first identified and named the Execution Order Inversion. They diagnose it as a
priority conflict: the GRO poll runs in a high-priority softirq, while
decryption runs in a normal-priority workqueue, so the poll repeatedly preempts
the workers that are trying to produce the very packets it needs. Their two
fixes both move GRO out of the softirq. The first re-schedules it onto a
dedicated workqueue running at normal priority, recovering a 4.7$\times$
throughput improvement. The second runs GRO in dedicated kernel threads, with a
similar effect. Both also cut tail latency substantially.

These fixes attack the \emph{cost} of each poll: by running GRO at the same
priority as decryption, a poll that finds nothing no longer steals time from
the workers. What they do not change is the \emph{number} of such polls. The
unconditional trigger in \texttt{wg\_queue\_enqueue\_per\_peer\_rx} still fires
after every decrypted packet, so a poll running while the head is not ready
still does no useful work. Moving that work to a cheaper context makes each
empty poll less harmful, but the empty polls remain.

Our work addresses the other side. We identify the unconditional trigger as the
root cause and eliminate empty polls at their source by checking whether the
head is ready before scheduling the poll at all. The two fixes are orthogonal:
one lowers cost per poll, the other lowers poll count. They can be applied
together.

\section{Root Cause Analysis}
\label{sec:rootcause}
%% ============================================================

Three conditions interact to produce the overhead: delivery is gated by the
queue head, decryption finishes out of order across cores, and the GRO trigger
checks neither before firing.

\subsection{Delivery Is Gated by the Head}

Each packet carries a state (\texttt{PACKET\_CB(skb)->state}, an
\texttt{atomic\_t}): \textsc{uncrypted} on enqueue, \textsc{crypted} on
success, \textsc{dead} on failure. \texttt{wg\_packet\_rx\_poll} walks from the
head and delivers while state is not \textsc{uncrypted}, halting at the first
\textsc{uncrypted} entry (\texttt{receive.c:451}). No packet can be delivered while the head
is still pending, regardless of how many later packets are ready.

\subsection{Decryption Finishes Out of Order}

Consecutive packets from one peer are not decrypted together. The device ring
assigns each to a core by round-robin (\texttt{queueing.h:120}), so they
decrypt in parallel and complete in an order determined by per-core load, not
arrival. Figure~\ref{fig:eoi} illustrates: a later packet finishes first and
schedules a poll, but the head is still pending.

\begin{figure}[t]
  \centering
  \includegraphics[width=0.92\linewidth]{report_eoi.pdf}
  \caption{Execution Order Inversion. Packet~2 finishes first and schedules a
  poll, but the head (packet~0) is still \textsc{uncrypted}, so the poll returns
  \texttt{work\_done = 0}. Packet~2 cannot be delivered until the head is
  resolved.}
  \label{fig:eoi}
\end{figure}

\subsection{The Trigger Ignores Both}

When a worker finishes, it marks the state and calls \texttt{napi\_schedule}
unconditionally (\texttt{queueing.h:195-196}):

\begin{lstlisting}[language=C]
atomic_set_release(&PACKET_CB(skb)->state, state);
napi_schedule(&peer->napi);
\end{lstlisting}

Each of $N$ parallel completions raises a poll, but only the one that makes the
head ready delivers anything. Every other completion produces a
\texttt{work\_done = 0} poll. The decrypt worker then proceeds to the next
packet and calls \texttt{napi\_schedule} again, so the wasted polls recur for
as long as the head lags behind.

\subsection{Why the Waste Grows with Core Count}

A poll is useful only at the moment the head becomes ready. Among $N$ packets
decrypting concurrently, and with no reason for queue position to bias
completion order, the head wins with probability $\approx 1/N$. As $N$ grows,
more siblings finish ahead of the head, and a larger fraction of polls find it
still \textsc{uncrypted}. The trigger fires once per decrypted packet; progress
is gated by a single event. The more cores, the more wasted polls. We measure
the actual rate directly in Section~\ref{sec:eval}.

%% ============================================================
\section{Proposed Fix}
\label{sec:fix}
%% ============================================================

\subsection{Condition Design}

The fix schedules the poll only when the front of the per-peer queue is no
longer \textsc{uncrypted}. If the front is still pending the wake is skipped;
whichever worker eventually makes the front ready will wake the poll then.

Reading the front safely from worker context requires care. The consumer-side
accessor \texttt{wg\_prev\_queue\_peek} caches its result and must not be
called from a worker. Instead we read the consumer cursor directly: in this
multi-producer single-consumer queue, \texttt{peer->rx\_queue.tail} is written
only by the consumer (\texttt{queueing.c:87,92,101}), so
\texttt{READ\_ONCE(peer->rx\_queue.tail)} is safe as a hint. If \texttt{tail}
points to the stub sentinel the worker schedules unconditionally, costing at
most one harmless extra poll.

\subsection{The Change}

Six lines in \texttt{queueing.h}, replacing the unconditional call:

\begin{lstlisting}[language=C]
/* was: napi_schedule(&peer->napi); */
tail = READ_ONCE(peer->rx_queue.tail);
if (tail == (struct sk_buff *)
            &peer->rx_queue.empty ||
    atomic_read(&PACKET_CB(tail)->state) !=
            PACKET_STATE_UNCRYPTED)
        napi_schedule(&peer->napi);
\end{lstlisting}

\subsection{Correctness}

Two things must hold: the guard never drops a needed wake, and the relaxed read
is sound.

Take the moment a worker marks its packet and evaluates the guard. If that
packet is the front, the front is now non-\textsc{uncrypted}, the guard passes,
and the poll runs and finds work. If the front is already \textsc{crypted} or
\textsc{dead} the guard passes. If the front is still \textsc{uncrypted} the
guard correctly skips the wake.

The one subtlety is a stale read. Suppose the front becomes ready on another
core at the same instant this worker reads \texttt{tail} and still sees
\textsc{uncrypted}. This worker skips the wake, but the core that flipped the
front to \textsc{crypted} runs the same guard and sees it ready, so it
schedules the poll. No packet is stranded. This is also why the read is safely
relaxed: it is a hint about whether to wake, not a synchronization point.

If a schedule lands while a poll is already running, NAPI records a
\texttt{MISSED} bit and re-runs the poll once when the current one finishes
(\texttt{dev.c:6744,6825}). No delivery is lost; at worst it is deferred by
one poll.

\subsection{Residual Limitation}

The guard does not reduce empty polls to exactly zero. The reason is the
\texttt{MISSED} mechanism that makes it safe. Suppose packets~0 and~1 finish
decrypting at almost the same time:
\begin{enumerate}
  \item The worker for packet~0 sees the front ready and wakes the poll, which
    starts running.
  \item The poll delivers packet~0, then packet~1 (also ready), and stops at
    packet~2, which is still \textsc{uncrypted}.
  \item The worker for packet~1 also saw the front ready and called
    \texttt{napi\_schedule}, but the poll was already running, so this only set
    \texttt{MISSED}.
  \item When the poll finishes, \texttt{MISSED} is set, so NAPI re-runs it.
    Packet~1 is already delivered and packet~2 is not ready, so this extra run
    finds nothing.
\end{enumerate}
So one empty poll can remain per such overlap. The key difference from the
unmodified code is scale: before, every out-of-order completion caused an empty
poll, and the count grew with the number of cores; now, the only empty polls
left are these occasional \texttt{MISSED} re-runs, which do not grow with
concurrency. We report the residual rate in the evaluation.

\section{Evaluation}
\label{sec:eval}

\subsection{Experimental Setup}

\textbf{Platform.} Apple M1~Pro (ARM64, 10 cores), Fedora Asahi Remix~42,
kernel \texttt{6.19.13-400.asahi.fc44.aarch64+16k}. The patched module exposes
sysfs knobs: \texttt{wg\_eoi\_fix} (0\,=\,stock, 1\,=\,fix) and
\texttt{wg\_decrypt\_delay\_us} (artificial per-packet busy-wait). Stock and
patched are compared within the same module instance and tunnel, eliminating
variance from module reloads.

\textbf{Topology and traffic.} One server namespace and $N$ client namespaces,
each a separate WireGuard peer with its own key pair. iperf3 drives a fixed
total of 32 streams split across $N$ peers ($\lfloor 32/N \rfloor$
streams/peer), holding offered load constant across the peer sweep. Each cell
runs 30~s with a 5~s warm-up omitted.

\textbf{Variance control and metrics.} Each (peers, build) cell is measured
5~times; stock and patched alternate order each run to cancel drift. GRO poll
counts and batch size (\texttt{work\_done}) are collected with bpftrace probes
on \texttt{wg\_packet\_rx\_poll}. Latency is from 20~Hz ping (p50/p99/p99.9).

\subsection{Discussion}

\textbf{RQ1} (mechanism): The unconditional \texttt{napi\_schedule} at
\texttt{queueing.h:196} fires after every decrypted packet;
\texttt{wg\_packet\_rx\_poll} returns \texttt{work\_done = 0} whenever the head
is \textsc{uncrypted}. This happens systematically with $N > 1$ concurrent
workers, which is the common case.

\textbf{RQ2} (prior fix): The prior fix changes GRO's scheduling context but
not the trigger. Wasted invocations occur at the same rate under their fix.

\textbf{RQ3} (conditional guard): The guard reduces wasted polls by 9--22\% in
the peer sweep and 20\% in the regime test. Batch size increases in both cases,
confirming that scheduling is deferred to points where delivery is productive.

\textbf{RQ4} (ARM): The EoI manifests on ARM with 25--31\% of GRO invocations
wasted under stock. The patch code is structurally identical between
\texttt{AsahiLinux/linux}~v6.19 and \texttt{torvalds/linux}~v6.1.

\subsection{The Cost of Each Wake}
\label{subsec:costwake}

The guard fixes \emph{when} to wake the poll, but leaves a second question
open: whether each wake is worth its cost. Because the guard wakes the poll as soon as the front is ready, a poll may
deliver as little as one packet. In that case WireGuard pays the full cost of a
softirq, a GRO pass, and a NAPI completion to hand up a single packet, with
none of GRO's batching benefit.

Whether this matters depends on two unmeasured costs: the overhead of a poll
itself, and the cost of delivering and copying a packet to the receiving
application. If delivery dominates, waking for one packet is fine. If poll
overhead dominates, a policy that waits for more packets to become ready before
waking would be better. We chose the simple wake-on-ready policy because it
keeps latency low and never delays a ready packet, which is the right trade-off
for a VPN. Measuring these two costs on real hardware is therefore central to
judging the fix long-term: it determines whether the simple policy is good
enough, or whether a batching-aware trigger is needed. We treat this as future
work.

%% ============================================================
\section{Conclusion}
\label{sec:conclusion}
%% ============================================================

We presented a source-level analysis of the Execution Order Inversion in
WireGuard's receive pipeline and a targeted fix for its root cause. The
analysis shows that the EoI stems from an unconditional \texttt{napi\_schedule}
call after every decrypted packet, which triggers GRO polls that systematically
find nothing to deliver when the per-peer queue head is still
\textsc{uncrypted}. The fraction of wasted polls grows with core count.

The fix is a six-line conditional guard that reads
\texttt{READ\_ONCE(peer->rx\_queue.tail)} and skips \texttt{napi\_schedule}
when the head is still pending. We proved correctness under stale reads and the stub sentinel edge
case. Experimental evaluation on ARM (Apple M1~Pro) confirms a 9--22\%
reduction in wasted GRO polls across peer counts 1--32 at fixed total load, and
a 20\% reduction under artificial heavy-crypto conditions with 2-core
contention. Batch size increases in both cases, confirming the mechanism.
Throughput is unchanged on this loopback rig, as expected: the benefit appears
in poll-efficiency metrics, not Gbps, because the M1 loopback does not saturate
\texttt{NET\_RX\_SOFTIRQ}.

The fix by \citeauthor{mounah2025} reduces the cost of each GRO invocation;
ours reduces how many invocations happen. The two are independent, and
applying both together should give additional improvement over either alone.

\smallskip\noindent\textbf{Future work.}
(1)~Validate the throughput effect on x86-64 hardware with a real 25~Gbps NIC
(CloudLab), the regime in which \citeauthor{mounah2025} measured.
(2)~Combine the conditional guard with the paper's workqueue-priority fix.
(3)~Measure wake cost vs.\ packet-delivery cost, then design a batching-aware
trigger policy that wakes only when waking pays off.
(4)~Extend the analysis to the handshake path, which uses a separate blocking
workqueue and may exhibit related overheads.
