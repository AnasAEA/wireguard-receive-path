\begin{abstract}
%% ============================================================

WireGuard is a modern VPN built into the Linux kernel. Under sustained
multi-gigabit traffic with 1,000 concurrent clients, it reaches only 19.2\% of
available bandwidth. Prior work traced this to an Execution Order Inversion
(EoI): concurrent decryption triggers GRO reassembly so frequently that the CPU
saturates. That work moved reassembly to a background thread and recovered
4.7$\times$ throughput.

We show, through source-level analysis, that this fix is incomplete. It makes
each wasted reassembly cheaper, but not less frequent. WireGuard schedules
reassembly after \emph{every} decrypted packet, even when no progress is
possible. With decryption spread over $N$ cores, the packet due for delivery
next is rarely the first to finish, so most reassembly passes find nothing to
deliver. We propose a six-line conditional guard: reassembly is triggered only
when the next packet due is ready. We prove correctness under concurrent access
and evaluate the fix on an ARM platform (Apple M1~Pro).

\end{abstract}

\section{Introduction}
%% ============================================================

WireGuard~\cite{donenfeld2017} is a modern VPN integrated into the Linux kernel
since version~5.6 (2020). It uses ChaCha20-Poly1305 encryption~\cite{rfc8439}
and the Noise protocol framework~\cite{noise}. At multi-gigabit throughput,
\citeauthor{mounah2025}~\cite{mounah2025} measured a throughput collapse to
19.2\% of line rate and 94\% CPU utilization under 25~Gbps with 1,000 clients.
They identified an Execution Order Inversion (EoI) and proposed moving GRO from
a high-priority softirq to a normal-priority workqueue, recovering
4.7$\times$ throughput.

Despite this result, the fix is incomplete. Moving GRO lowers the cost of each
wasted poll but does not lower their count. The root cause is unchanged:
WireGuard calls \texttt{napi\_schedule} after every decrypted packet regardless
of queue state. GRO can only deliver if the \emph{head} of the per-peer queue is
already decrypted. With $N$ packets decrypting concurrently the head finishes
first with probability $1/N$, so on average $(N\!-\!1)/2$ polls per batch find
nothing to deliver. This is a trigger logic problem, not a scheduling priority
problem.

This work presents a source-level analysis and a targeted fix: a six-line guard
at the scheduling site that invokes GRO only when the head is ready. We prove
the guard correct under concurrent access and evaluate it experimentally.

\smallskip\noindent\textbf{Research questions.}
\textbf{RQ1:} What exactly causes GRO to be invoked wastefully?
\textbf{RQ2:} Does the prior fix eliminate the root cause or only reduce
per-invocation cost?
\textbf{RQ3:} Can a conditional guard eliminate the dominant source of wasted
invocations without breaking ordering?
\textbf{RQ4:} Does the problem manifest on ARM as on x86-64?

\smallskip\noindent\textbf{Scope.} We focus on the data receive path.
Measurements are on ARM (Apple M1~Pro, Fedora Asahi Remix); x86-64 validation
on dedicated server hardware (CloudLab) is in progress.

\section{Background}
\label{sec:background}
%% ============================================================

\subsection{WireGuard's Receive Pipeline}

WireGuard's receive path has three stages. Stage~1 runs in bottom-half (BH)
context: it de-encapsulates incoming UDP datagrams and dispatches them for
decryption. Stage~2 performs ChaCha20-Poly1305 decryption inside a
\emph{workqueue}, a pool of kernel threads running at normal scheduler
priority. Stage~3 reassembles decrypted packets and passes them to the network
stack through Generic Receive Offload (GRO), executing as a \emph{softirq}.

Two separate queues serve different purposes (Figure~\ref{fig:queues}): the
\emph{per-peer RX queue} holds one peer's packets in arrival order to fix
\emph{delivery order}; the \emph{global device ring} distributes packets across
CPU cores by round-robin to pick \emph{which core} decrypts each one. Delivery
order is fixed first, when the packet enters the per-peer queue with state
\textsc{uncrypted}. When a worker finishes, it flips the state to
\textsc{crypted} (or \textsc{dead} on failure).

\begin{figure}[t]
  \centering
  \includegraphics[width=\linewidth]{report_two_queues.pdf}
  \caption{The same packet enters both queues for different reasons. The
  per-peer queue fixes \emph{delivery order}; the device ring picks \emph{which
  CPU} decrypts. Workers flip each packet's state; \texttt{wg\_packet\_rx\_poll}
  delivers from the head in order.}
  \label{fig:queues}
\end{figure}

\subsection{NAPI, GRO, and Softirqs}

NAPI batches packet delivery: after the first packet, the kernel stops
per-packet interrupts and instead polls the device in batches. Each receive
queue has a \texttt{napi\_struct} and a poll function; WireGuard assigns each
peer its own, with \texttt{wg\_packet\_rx\_poll} as its poll function.
Calling \texttt{napi\_schedule} sets a ``scheduled'' flag and raises
\texttt{NET\_RX\_SOFTIRQ}; it does not run the poll immediately. Repeated calls
while the flag is set collapse into a single pending poll.

When the softirq runs, \texttt{wg\_packet\_rx\_poll} walks the per-peer queue
from its head and passes packets to GRO, which merges consecutive packets of the
same flow to amortize the stack-traversal cost. The poll stops at the first
\textsc{uncrypted} entry and handles at most 64 packets (the budget).
\texttt{napi\_complete\_done} then clears the flag.

\subsection{Linux Workqueues}

WireGuard's decrypt workqueue (\texttt{packet\_crypt\_wq}) uses
\texttt{WQ\_PERCPU}, provisioning one worker per online core. Work items are
dispatched to a specific core via \texttt{queue\_work\_on}. Workers run at
\texttt{SCHED\_NORMAL} and can be preempted by softirqs at any BH re-enable
point.

\section{Related Work}
\label{sec:sota}
%% ============================================================

\citeauthor{mounah2025}~\cite{mounah2025} first identified the EoI, diagnosing
it as a priority conflict: the GRO poll runs at high softirq priority and
repeatedly preempts the workers producing the packets it needs. Their two fixes
both move GRO out of the softirq --- one onto a dedicated workqueue, one into
kernel threads --- recovering 4.7$\times$ throughput and cutting tail latency.

These fixes lower the \emph{cost} of each poll but not their \emph{count}. The
unconditional trigger in \texttt{wg\_queue\_enqueue\_per\_peer\_rx} is
unchanged, so polls still fire after every completion whether or not the head is
ready. Our work addresses this other half: checking head readiness before
scheduling eliminates empty polls at their source. The two fixes are orthogonal
and can be combined.

%% ============================================================
\section{Root Cause Analysis}
\label{sec:rootcause}
%% ============================================================

The overhead arises from three interacting facts.

\subsection{Delivery Is Gated by the Head}

Each packet carries a state (\texttt{PACKET\_CB(skb)->state}, an
\texttt{atomic\_t}): \textsc{uncrypted} on enqueue, \textsc{crypted} on
success, \textsc{dead} on failure. \texttt{wg\_packet\_rx\_poll} walks from the
head and delivers while state is not \textsc{uncrypted}, halting at the first
\textsc{uncrypted} entry (\texttt{receive.c:451}). The invariant is absolute:
no packet can be delivered while the head is still pending, no matter how many
later packets are ready.

\subsection{Decryption Finishes Out of Order}

Consecutive packets from one peer are not decrypted together. The device ring
assigns each to a core by round-robin (\texttt{queueing.h:120}), so they
decrypt in parallel and complete in an order set by per-core load, not arrival.
Figure~\ref{fig:eoi} shows the result: a later packet finishes first and
schedules a poll, but the head is still pending.

\begin{figure}[t]
  \centering
  \includegraphics[width=0.92\linewidth]{report_eoi.pdf}
  \caption{Execution Order Inversion. Packet~2 finishes first and schedules a
  poll, but the head (packet~0) is still \textsc{uncrypted}, so the poll returns
  \texttt{work\_done = 0}.}
  \label{fig:eoi}
\end{figure}

\subsection{The Trigger Ignores Both}

When a worker finishes, it calls \texttt{napi\_schedule} unconditionally
(\texttt{queueing.h:195-196}):

\begin{lstlisting}[language=C]
atomic_set_release(&PACKET_CB(skb)->state, state);
napi_schedule(&peer->napi);
\end{lstlisting}

Each of $N$ parallel completions raises a poll, but only the one that makes
the head ready delivers anything. Every other completion produces a
\texttt{work\_done = 0} poll.

\subsection{Why the Waste Grows with Core Count}

Among $N$ packets decrypting concurrently, the head finishes first with
probability $\approx 1/N$. As $N$ grows, more siblings complete ahead of the
head, and a larger fraction of polls find it still \textsc{uncrypted}. The
trigger fires once per decrypted packet; a poll makes progress only at the head.
The more cores, the more wasted polls. We measure the actual rate in
Section~\ref{sec:eval}.

%% ============================================================
\section{Proposed Fix}
\label{sec:fix}
%% ============================================================

\subsection{Condition Design}

The fix schedules the poll only when the front of the per-peer queue is no
longer \textsc{uncrypted}. If the front is still pending the wake is skipped;
the worker that eventually makes the front ready will wake the poll then.

Reading the front safely from worker context requires care. The consumer-side
accessor \texttt{wg\_prev\_queue\_peek} caches its result and must not be called
from a worker. Instead we read the consumer cursor directly: in this
multi-producer single-consumer queue, \texttt{peer->rx\_queue.tail} is written
only by the consumer (\texttt{queueing.c:87,92,101}), so
\texttt{READ\_ONCE(peer->rx\_queue.tail)} is safe as a hint. If \texttt{tail}
points to the stub sentinel the worker schedules unconditionally, costing at
most one harmless extra poll.

\subsection{The Change}

Six lines in \texttt{queueing.h}, replacing the unconditional call:

\begin{lstlisting}[language=C]
/* was: napi_schedule(&peer->napi); */
tail = READ_ONCE(peer->rx_queue.tail);
if (tail == (struct sk_buff *)
            &peer->rx_queue.empty ||
    atomic_read(&PACKET_CB(tail)->state) !=
            PACKET_STATE_UNCRYPTED)
        napi_schedule(&peer->napi);
\end{lstlisting}

\subsection{Correctness}

If the finishing packet is the front, the front is now non-\textsc{uncrypted},
the guard passes, and the poll finds work. If the front is already
\textsc{crypted}/\textsc{dead} the guard passes. If the front is still
\textsc{uncrypted} the guard correctly skips the wake. For stale reads: if this
worker misses a concurrent flip to \textsc{crypted}, the core that made that
flip runs the same guard and schedules the poll, so no packet is stranded. The
read is safely relaxed (\texttt{atomic\_read}): it is a hint, not a
synchronization point.

If a schedule lands while a poll is running, NAPI records a \texttt{MISSED}
bit and re-runs the poll once (\texttt{dev.c:6744,6825}). No delivery is lost;
at worst it is deferred by one poll.

\subsection{Residual Limitation}

The \texttt{MISSED} re-run can itself be empty. If two workers both see the
front ready, both call \texttt{napi\_schedule}; the second only sets
\texttt{MISSED}. When the poll finishes and re-runs on the \texttt{MISSED}
signal, the front may already have been delivered, leaving nothing to do. These
occasional re-runs do not grow with core count --- unlike the unmodified code,
where every out-of-order completion produced an empty poll.

\section{Evaluation}
\label{sec:eval}
%% ============================================================

\subsection{Experimental Setup}

\textbf{Platform.} Apple M1~Pro (ARM64, 10 cores), Fedora Asahi Remix~42,
kernel \texttt{6.19.13-400.asahi.fc44.aarch64+16k}. The patched module exposes
sysfs knobs: \texttt{wg\_eoi\_fix} (0 = stock, 1 = fix) and
\texttt{wg\_decrypt\_delay\_us} (artificial per-packet busy-wait). Stock and
patched are compared within the same module instance and tunnel, eliminating
variance from module reloads.

\textbf{Topology and traffic.} One server namespace and $N$ client namespaces,
each a separate WireGuard peer. iperf3 drives a fixed total of 32 streams split
across $N$ peers ($\lfloor 32/N \rfloor$ streams/peer), so offered load is
constant across the peer sweep. Each cell runs 30~s with a 5~s warm-up omitted.

\textbf{Variance control and metrics.} Each (peers, build) cell is measured
5~times; stock and patched alternate order each run. GRO poll counts and batch
size (\texttt{work\_done}) are collected with bpftrace probes on
\texttt{wg\_packet\_rx\_poll}. Latency from 20~Hz ping (p50/p99/p99.9).

\subsection{Discussion}

\textbf{RQ1}: Confirmed. \texttt{napi\_schedule} fires unconditionally at
\texttt{queueing.h:196}; \texttt{wg\_packet\_rx\_poll} returns
\texttt{work\_done = 0} whenever the head is \textsc{uncrypted} ---
systematically with $N > 1$ concurrent workers.

\textbf{RQ2}: The prior fix changes GRO's scheduling context; it does not
change the trigger. Wasted invocations occur at the same rate under their fix.

\textbf{RQ3}: Confirmed. The guard reduces wasted polls by 9--22\% in the
sweep and 20\% in the regime test. Batch size increases in both cases.

\textbf{RQ4}: EoI manifests on ARM with 25--31\% of GRO invocations wasted
under stock. The patch code is structurally identical between
\texttt{AsahiLinux/linux}~v6.19 and \texttt{torvalds/linux}~v6.1.

\subsection{The Cost of Each Wake}

The guard wakes the poll as soon as the front is ready, so a poll may deliver
just one packet. Whether this is a problem depends on two unmeasured costs: the
overhead of a poll vs.\ the cost of delivering and copying a packet to
userspace. If delivery dominates, waking for one packet is fine; if poll
overhead dominates, a batching-aware policy would be better. We chose the
simple wake-on-ready policy because it keeps latency low and never delays a
ready packet --- the right trade-off for a VPN. The better fix, waking only
when waking pays off, requires these measurements and is future work.

%% ============================================================
\section{Conclusion}
\label{sec:conclusion}
%% ============================================================

WireGuard's EoI stems from an unconditional \texttt{napi\_schedule} after every
decrypted packet: GRO polls systematically find nothing to deliver when the
per-peer queue head is still \textsc{uncrypted}, and the fraction of such polls
grows with core count. A six-line conditional guard --- reading
\texttt{READ\_ONCE(peer->rx\_queue.tail)} before waking the poll --- fixes the
trigger. We proved correctness under stale reads and the stub sentinel edge
case. Evaluation on ARM confirms a 9--22\% reduction in wasted polls across
peer counts 1--32, with batch size increasing in all cases. Throughput is
unchanged on this loopback rig, as expected; the GRO improvement appears in
poll-efficiency metrics, not Gbps.

The fix by \citeauthor{mounah2025} (lower cost per poll) and our fix (lower
poll count) are orthogonal; combining them should yield additive benefit.
Validating this on x86-64 server hardware with a 25~Gbps NIC (CloudLab) is the
primary next step, alongside measuring wake cost vs.\ packet-delivery cost to
guide a batching-aware trigger policy.
