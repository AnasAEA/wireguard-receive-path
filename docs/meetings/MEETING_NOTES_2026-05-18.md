## Thursday deliverable — EoI proof from source code

Alain does not fully trust the Mounah et al. paper as a standalone argument. He wants to see EoI proven from first principles — code-level evidence that the inversion actually exists, not just a citation.

The presentation must walk through the exact call chain in the WireGuard source and show, line by line, why the priority inversion happens and why it self-reinforces.

### The chain to present (already traced)

1. `wg_packet_decrypt_worker` — `receive.c:492` — runs at `SCHED_NORMAL` inside `packet_crypt_wq`
2. After each packet, calls `wg_queue_enqueue_per_peer_rx` → `napi_schedule(&peer->napi)` — `queueing.h:196`
3. `napi_schedule` does two things: marks GRO as a pending softirq AND records the current CPU into the napi_struct — stale pointer, not a live load query
4. When `ptr_ring_consume_bh` releases BH, any pending softirq fires immediately — GRO preempts the decryption worker at high priority
5. GRO finds no completed packets, aborts
6. On the next burst, GRO is still pinned to the same recorded CPU — workers have migrated but the pointer hasn't updated — that core saturates at 94%, others sit at 20%
7. The imbalance self-reinforces on every packet burst

### What this proves without citing the paper

- The preemption is not accidental — it is a direct consequence of `spin_unlock_bh` semantics and `napi_schedule` recording behavior
- The self-reinforcing saturation is a structural property of `napi_schedule`'s CPU pinning, not a load anomaly
- The fix (moving GRO to a workqueue) works because it eliminates the softirq/workqueue priority gap — GRO and decryption would then run at the same priority level

## 1. Status of the proxy argument

The proxy argument (io-wq as a proxy for WireGuard's workqueue) is no longer valid.

Source-level analysis of `io-wq.c` confirms that io-wq uses kernel threads created via `create_io_thread`, woken via `wake_up_process`. The `work_struct` appears in io-wq only as a fallback mechanism (`INIT_DELAYED_WORK`) for worker creation retry — not as the primary dispatch path. This is a rescue system, not the normal packet path.

WireGuard and io-wq use structurally different infrastructures. The proxy claim does not hold at implementation level.

---

## 2. What the two systems do share

Despite the implementation difference, WireGuard (kernel) and wireguard-go share the same **three-stage pipeline structure**:

| Stage | Kernel WireGuard | wireguard-go |
|---|---|---|
| 1 | UDP receive handler | RoutineReceiveIncoming goroutine |
| 2 | Decryption workqueue (packet_crypt_wq) | RoutineDecryption goroutines (NumCPU) |
| 3 | GRO/NAPI softirq | RoutineSequentialReceiver goroutine |

**EoI is impossible in wireguard-go** because Stage 3 (RoutineSequentialReceiver) blocks on a mutex that Stage 2 (RoutineDecryption) holds. Stage 3 cannot run until Stage 2 releases the lock. There is no softirq that can preempt a goroutine mid-execution from userspace. The ordering is enforced by construction.

## 4. The new objective — agreed in the meeting

The core question becomes:

> **Do the work items generated by WireGuard's workqueue call blocking functions — and if so, what is the impact on scheduling latency and worker throughput?**

If a work item calls a blocking function (synchronization primitive, blocking I/O, synchronous crypto library call), this can introduce latency that affects work creation and management, increases pressure on worker threads, and degrades other workers in the pool.

### Concrete steps

1. **List all work items created by WireGuard** — identify every `queue_work_on` call and the associated worker function
2. **For each work item, identify the functions it calls** — trace the full call chain
3. **Check for blocking behavior** — locks, blocking syscalls, synchronous crypto operations (ChaCha20-Poly1305, BLAKE2)
4. **If blocking functions are found:** propose mitigations (e.g., grouping work items at normal priority to avoid conflicting interrupts, batching)
5. **If no blocking functions are found:** conclude the problem does not exist in this form, and redirect

### Measurement plan

- Analyze io_uring-generated work items in the same way, to compare behavior
- Verify work item behavior using kernel traces
- If needed, build a micro-benchmark to illustrate impact on latency and thread consumption
